Conflict

Apa itu Conflict?

Asumsikan Anda bekerjasama dengan saya pada suatu proyek. Suatu ketika, saya mengedit sebuah halaman yang sudah tercantum di GitHub dan demikian juga anda. Kita sama-sama mengedit halaman yang sama dan anda sudah mem-push hasil pekerjaan anda duluan. Baru setelah itu saya selesai dan saya push.

Tiba-tiba saya kaget, "Wah, kok saya dapet pesan Conflict nih?"

Ketika saya cek di repository GitHub, saya melihat anda baru saja push ke repository dan saya maklum apabila terjadi sebuah conflict. Karena sifat Git yang memiliki integritas, dia akan "memantau" secara keseluruhan kode yang berada di GitHub dan dibandingkan dengan kode yang hampir saya push. Ternyata kode kita sangatlah berbeda! Dari segi penulisan maupun algoritma yang digunakan.

Penyebab Conflict

Penyebab conflict adalah adanya ketidaksesuaian data dari lokal dengan data yang ada pada server, data saling clash satu sama lain sehingga menyebabkan conflict. Sebagai contoh, saya dan anda melakukan revisi pada baris yang sama dengan kode yang jauh berbeda. Maka Git yang mencoba untuk melakukan Auto Merging tidak berhasil karena baris yang sama dimasukkan code yang berbeda sehingga menimbulkan conflict.

Menghindari Conflict

Untuk menghindari conflict, bisa melakukan git pull terlebih dahulu sebelum mengedit bagian yang lain. Namun cara ini kurang efektif karena pada akhirnya kita terkesan harus menunggu sebelum kita bisa mulai mengedit bagian lain. Dan ini tidak mungkin dilakukan apabila pekerjaan kita diharuskan untuk kerja cepat.

Oleh sebab itu, kita bisa mengakali dengan membuat branch baru pada Git untuk menyimpan sementara kode masing-masing sebelum nanti digabungkan dengan cara git merge ke branch master.

Mengatasi Conflict

Apabila sudah terlanjur menghadapi sebuah conflict git, maka yang harus kita lakukan adalah menyimpan kode sementara yang kita buat. Lalu lakukan git pull terlebih dahulu sehingga kita akan mendapatkan kode yang paling terupdate. Barulah setelah itu kita edit dan tambahkan kode yang sudah kita buat sebelumnya. Lalu bisa kita push langsung ke master ataupun kita push dulu ke branch baru. Push ke branch baru lebih disarankan untuk menghindari error lainnya. Setelah itu, barulah kita lakukan git merge ke branch master.
